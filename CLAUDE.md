# CLAUDE.md — agent-maker 작업 지침

> 이 파일은 **agent-maker 저장소에서 작업하는 Claude Code를 위한 지침**이다.
> agent-maker가 *생성해 내보내는* 챗봇용 `CLAUDE.md`와 혼동하지 말 것. (그건 산출물 템플릿이다.)

---

## 0. 가장 먼저 읽을 것

작업을 시작하기 전에 **반드시 [`PLAN.md`](PLAN.md)를 읽는다.** PLAN.md가 이 프로젝트의
단일 진실 공급원(SSOT)이다. 설계·범위·로드맵에 대한 질문의 답은 거의 다 거기에 있다.

설계 결정이 바뀌면 **코드보다 PLAN.md를 먼저 수정**한다.

---

## 1. 이 프로젝트가 무엇인가

**agent-maker** 는 공공기관 납품용 챗봇 에이전트를 구성하기 위한 **시각적 설정 마법사(웹앱)** 다.
사용자가 테마·스택·RAG·LLM·API·컴플라이언스 등 모든 요소를 보면서 선택하면,
**Claude Code가 한 번에 챗봇을 구현할 수 있는 산출물 묶음(ZIP)** 을 만들어 준다.

> 한 문장: **"챗봇을 만드는 도구가 아니라, 챗봇을 만들기 위한 설계도+지시서를 만드는 도구."**

상세는 PLAN.md §1~§7 참조.

---

## 2. 기술 스택

- **Next.js (App Router)** + React 19 + **TypeScript (strict)**
- **Tailwind CSS** (테마 토큰/CSS 변수 기반)
- **Zod** — `AgentSpec` 스키마 = 검증 = 타입의 SSOT
- 상태 관리 / UI 라이브러리 / ZIP 라이브러리는 PLAN.md §3·§9에서 확정 예정 (현재 후보 단계)

패키지 매니저는 **npm** 을 사용한다 (`package-lock.json` 커밋).

---

## 3. 디렉토리 구조 (의도)

```
agent-maker/
├─ PLAN.md              # ★ SSOT — 항상 먼저 읽기
├─ CLAUDE.md            # 이 파일
├─ README.md            # 사람용 개요/실행법
├─ docs/
│  └─ spec-schema.md    # AgentSpec 필드/enum 상세 정의
├─ .claude/
│  └─ skills/           # 프로젝트 전용 skill (반복 작업 자동화)
├─ src/
│  ├─ app/              # Next.js App Router (라우트/페이지)
│  ├─ components/       # UI 컴포넌트 (마법사 스텝, 미리보기 등)
│  ├─ lib/              # 스키마(AgentSpec), 상태, 유틸
│  ├─ catalog/          # 단계별 선택지 데이터 (LLM·임베딩·VectorDB 목록 등)
│  └─ generators/       # 산출물 생성기 (PROMPT/DESIGN/CLAUDE.md, zip)
└─ public/              # 정적 자산
```

> 구조는 구현이 진행되며 채워진다. 새 파일은 위 의도에 맞는 위치에 둔다.

**프로젝트 skill** (`.claude/skills/`) — 반복 작업용. 해당 작업을 할 때 활용하고, 부족하면 개선한다(§7):
- `catalog-add` — 새 선택지(LLM·임베딩·Vector DB 등)를 카탈로그에 추가
- `wizard-step` — 마법사 단계 추가/수정 (스키마↔타입↔UI↔생성기 정합성)
- `export-verify` — 생성된 산출물(ZIP)이 실제 Claude Code에서 동작하는지 E2E 검증(게이트)
- `docs-consistency` — PLAN/spec-schema/코드/README 정합성 점검

---

## 4. 핵심 설계 규칙 (반드시 지킬 것)

1. **선택지를 코드에 하드코딩하지 않는다.** LLM·임베딩·Vector DB 등 모든 목록은
   `src/catalog/` 의 **데이터**로 둔다. 새 옵션 추가 = 데이터 추가, 컴포넌트 수정 불필요.
2. **`AgentSpec`(Zod)이 단일 진실.** 폼 상태·검증·기본값·산출물은 모두 이 스키마에서 파생한다.
   필드를 추가하면 스키마 → 타입 → UI → 생성기 순으로 일관되게 반영한다.
3. **시각적 선택이 원칙.** 새 선택 UI는 텍스트 라디오가 아니라 **렌더된 미리보기**(스와치/카드/위젯)로 만든다. (PLAN.md §2·§4 Step 1)
4. **공공기관 제약은 1급 시민.** 폐쇄망/온프레미스, KWCAG 접근성, 개인정보, 데이터 국내 보관, 국산/오픈소스 우선을 항상 고려한다. 충돌(예: 폐쇄망 + 클라우드 임베딩 API)은 마법사가 **경고**해야 한다.
5. **산출물 품질이 제품의 본질.** 생성되는 `PROMPT.md`는 Claude Code가 **모호함 없이 즉시 구현 착수**할 수준이어야 한다. (PLAN.md §6)
6. **클라이언트 사이드 우선.** 산출물 생성·ZIP은 가능하면 브라우저에서 처리한다(폐쇄망 친화, 서버 의존 최소화).

---

## 5. 코딩 컨벤션

- TypeScript `strict`. `any` 회피, 가능한 한 스키마에서 추론한 타입 사용.
- 컴포넌트: 함수형, 작은 단위. 한 파일에 한 가지 책임.
- 명명: 컴포넌트 `PascalCase`, 함수/변수 `camelCase`, 상수/카탈로그 키 `SCREAMING_SNAKE` 또는 안정적 `kebab` id.
- 주석/문서 문자열은 **한국어**로 작성한다(이 프로젝트의 공용어). 코드 식별자는 영어.
- 스타일은 Tailwind 유틸리티 우선. 색은 직접 hex가 아니라 **테마 토큰/CSS 변수**로.
- 접근성: 시맨틱 HTML, `aria-*`, 키보드 내비게이션, 대비(KWCAG)를 기본값으로 만든다.

## 7. 작업 방식

- **PLAN.md의 로드맵(§8)을 따른다.** 현재 마일스톤을 벗어난 큰 작업을 하기 전엔 사용자에게 확인한다.
- 사용자 지시: **"바로 구현하지 말고 계획·세팅을 먼저"** — 큰 기능을 만들기 전 계획/문서를 갱신한다.
- **애매한 사항은 `AskUserQuestion`을 적극적으로 사용해 확인한다.** 추측해서 진행하지 말 것.
  스택/선택지/범위/우선순위 등 결정이 결과를 바꾸는 지점에서는 임의로 정하지 말고 질문으로 확인한다.
  (단, 명백한 관례나 코드에서 확인 가능한 사실은 질문 없이 진행하고 결과에 언급한다.)
- 결과를 정직하게 보고한다: 빌드/실행을 했으면 실제 결과를, 건너뛴 단계가 있으면 그렇다고 말한다.
- **커밋은 작업 단위로 하되, `git push`는 사용자가 명시적으로 요청할 때만 한다.** (2026-06-20 정책)
  메인 브랜치 직접 커밋은 사용자가 허용한 경우에만.
- **서브에이전트는 최소 Sonnet 이상 모델을 사용한다.** Task/Agent로 서브에이전트를 띄울 때는
  Haiku 등 하위 모델을 쓰지 말고, 반드시 **Sonnet 이상**(Sonnet / Opus 등)을 지정한다.
  (이 프로젝트의 작업은 설계·코드 품질이 중요하므로 추론 능력이 충분한 모델이 필요하다.)

### 지침·skill 지속 개선 (중요)

이 프로젝트의 지침(`CLAUDE.md`, `PLAN.md`, `docs/`)과 skill(`.claude/skills/`)은 **고정된 것이
아니라 개발이 진행되며 계속 다듬어 가는 살아있는 문서**다. 다음 상황에서 **먼저 멈추지 말고
해당 문서/스킬을 갱신**한 뒤(또는 갱신을 제안한 뒤) 작업을 잇는다.

- **반복 작업을 발견하면** → skill로 만들거나 기존 skill을 보강한다. (같은 절차를 3번쯤 반복하면 신호)
- **지침과 실제 코드/관례가 어긋나면** → 지침을 현실에 맞게 고친다. (지침이 낡으면 오히려 해롭다)
- **설계 결정이 바뀌면** → 코드보다 **PLAN.md를 먼저** 고친다. (CLAUDE.md §0)
- **새 결정·제약·합의가 생기면** → 적절한 문서(PLAN/spec-schema/CLAUDE)에 기록해 다음 세션이 알 수 있게 한다.
- **skill을 실제로 써보고 부족하면** → 그 skill을 즉시 개선한다. (skill은 쓰면서 자란다)

> 원칙: "다음에 또 할 일"이라면 머릿속이 아니라 **문서나 skill에 남긴다.** 큰 변경(지침의 방향을
> 바꾸는 수준)은 `AskUserQuestion`으로 확인하고, 작은 정합성 수정은 바로 반영한다.
